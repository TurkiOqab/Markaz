# تصميم: صلاحية "ربيع" (مدير شعبة العمليات) — المرحلة 1

**التاريخ:** 2026-05-17
**النطاق:** تمييز تسجيل الدخول + الصفحة الترحيبية لربيع فقط. لا backend، لا قاعدة بيانات.

---

## 0. خلفية: مطابقة الوصف مع الكود الفعلي

الوصف الأصلي افترض بنية غير موجودة في المشروع. الواقع المؤكَّد بعد فحص الكود:

- المصادقة جدول واحد `chiefs` (مدير المركز): `username` + `password_hash` فقط. **لا حقل `role`**. `/api/setup` يرفض أي حساب ثانٍ صراحةً (409).
- لا يوجد مفهوم "مراكز متعددة". الأقرب هو الفِرَق/الورديات `teams` بأسماء "أ"، "ب"، "ج" — وليست م22/م23.
- "التكميل اليومي" = جدول `roll_calls`، صفّ واحد لكل يوم لهذا المركز الواحد (تعداد أفراد الوردية)، `date` فريد. لا تكميل لكل مركز يُجمَّع لأعلى.
- الصفحة الترحيبية الحالية = Overlay داخل `LoginTransition.tsx` يعرض "مرحباً بعودتك إنجاز ٢٢".

التطبيق اليوم = **مركز واحد، مستخدم واحد، تكميل يومي واحد**. لذلك بُني هذا التصميم على بيانات تكميل ثابتة (mock) ومنطق زمني حقيقي، دون أي مساس بقاعدة البيانات أو بمسار إنجاز.

### قرارات المستخدم الموثَّقة

1. **مصدر بيانات بطاقة التكميل:** بيانات تجريبية ثابتة الآن (لا لمس لقاعدة البيانات). النموذج الحقيقي للمراكز يُبنى في جلسة لاحقة.
2. **آلية تمييز الدخول:** تحقق أمامي خاص بـ `REB9` (لا استدعاء backend). قبل المستخدم القيد الأمني (كلمة المرور مكتوبة في كود الواجهة) كحل مرحلة-أولى مؤقت لتطبيق محلي أحادي المستخدم.
3. **زر "الانتقال إلى لوحة التحكم":** صفحة "قيد التطوير" بنفس الهوية البصرية مع زر عودة.
4. **المقاربة المعمارية:** (ب) إعادة استخدام كوريوغرافيا Overlay الحالي — **بصيغة آمنة**: مشاركة أنيميشن CSS العام + نسخ القشرة البصرية في ملفات ربيع جديدة، و`LoginTransition.tsx` بلا أي تعديل.

## 2. المعمارية — "ب" بصيغة آمنة

### مبدأ العزل
- `frontend/src/components/LoginTransition.tsx` ومسار إنجاز: **صفر تعديل**. أي مستخدم ≠ `REB9` يسلك المسار الحالي حرفيًّا وبنفس السلوك.
- إعادة استخدام keyframes العامة في `frontend/src/index.css` (`animate-gold-sweep`, `animate-welcome-reveal`, `animate-welcome-leave`, `animate-fade-slide-*`, `animate-veil-sweep`, `animate-sheen-sweep`, `animate-shape-drift-slow`, `animate-status-pulse`) — مشتركة وغير خاصة بإنجاز.
- القشرة البصرية (خلفية الشعار، الخطوط الذهبية، البطاقات الزجاجية) **تُكرَّر عمدًا** في ملفات `rabea/` لحماية إنجاز. دَيْن تقني معروف يُوحَّد عند بناء لوحة ربيع الحقيقية لاحقًا.

### اللمستان الوحيدتان على كود قائم (لا مفر منهما لأي ربط دخول)
1. **`frontend/src/pages/LoginPage.tsx`** — شرط محروس في بداية `handleSubmit`:
   إذا `username === "REB9" && password === RABEA_PASSWORD` → تفعيل وضع ربيع (علَم sessionStorage) + تشغيل انتقال ربيع، ثم `return;` **قبل** استدعاء `login()` الخاص بالـ backend. فرع إنجاز (else / الباقي) يبقى كما هو دون تغيير سلوكي.
2. **`frontend/src/App.tsx`** — تركيب `RabeaWelcomeTransitionProvider` إضافةً (لا استبدالًا) لمزوّد إنجاز، وتسجيل مسار `/operations`.

## 3. آلية الدخول والتدفّق

- ثوابت في `rabea/rabeaSession.ts`: `RABEA_USERNAME = "REB9"`، `RABEA_PASSWORD = "1234567891"`، ومفتاح علَم الجلسة `injaz_rabea_mode` في `sessionStorage`.
- عند مطابقة بيانات `REB9`: ضبط العلَم، استدعاء `start()` لمزوّد انتقال ربيع، والعودة المبكرة من `handleSubmit`.
- كوريوغرافيا ربيع تطابق نمط إنجاز: دخول ~1000ms → ثبات (بانتظار المستخدم) → خروج ~500ms. زر CTA يبدأ الخروج ويوجّه إلى `/operations`.
- **انحراف موثَّق عن الوصف الأصلي:** الوصف ذكر مسارًا دائمًا `/operations-welcome`. لكن إعادة استخدام الكوريوغرافيا (قرار "ب") تجعل الترحيب Overlay لا مسارًا دائمًا — تمامًا كإنجاز. لذا لا يوجد مسار `/operations-welcome`؛ يوجد `/operations` فقط (قيد التطوير).
- حارس بسيط: `/operations` يتحقق من علَم `injaz_rabea_mode`؛ إن غاب → تحويل لـ `/login`.

## 4. محتوى الصفحة الترحيبية لربيع

- **الترويسة:** شعار `/logo.png` + "نظام إنجاز / لوحة شعبة العمليات" · ساعة حيّة بصيغة عربية (نفس منطق التنسيق في `LoginTransition`). تُحذف أزرار الإنذار (أبيض/برتقالي/أحمر) لأنها خاصة بعمليات إنجاز ولا يحتاجها ربيع في المرحلة 1.
- **العنوان:** «مرحباً بعودتك» + «إنجاز ٩.» (باللون الذهبي `injaz-gold-soft`، بنفس نمط "إنجاز ٢٢.").
- **سطر فرعي:** `<التاريخ الطويل لليوم ديناميكيًّا> · ربيع - مدير شعبة العمليات`. (التاريخ في الوصف "8 مايو" مثال فقط؛ يُستخدم تاريخ اليوم الفعلي.)
- **CTA:** زر ذهبي «الانتقال إلى لوحة التحكم →» مفعّل دائمًا (بخلاف إنجاز الذي يشترط اختيار فرقة).
- **التخطيط:** عمودان (RTL) — النصوص الترحيبية يمينًا، بطاقة التكميل يسارًا. على الشاشات الصغيرة: البطاقة فوق النصوص.

## 5. بطاقة «حالة التكميل اليومي» (العنصر الرئيسي)

### العرض
- أيقونة 📋 + عنوان "حالة التكميل اليومي".
- رقم كبير `X / 2` + شريط تقدّم ملوّن + نسبة مئوية.
- صفّ لكل مركز (م22، م23) بحالته.

### البيانات والمنطق
- **بيانات ثابتة (mock)** في `rabea/takmeelMock.ts`: دالة `getTodayTakmeel(): CenterTakmeel[]` تُعيد مثلًا `[{ id: "م22", submittedAt: "07:32" }, { id: "م23", submittedAt: null }]`. هذه هي نقطة الاستبدال المستقبلية بـ API حقيقي.
- **عتبة زمنية حقيقية:** الموعد الرسمي 09:00 ص. منطق التأخير يُحسب مقابل الساعة الفعلية.
- **دالة نقية** `rabea/takmeelView.ts` → `deriveTakmeelView(centers, now)` تُنتج كل الحالات:
  - **قبل 09:00:** حالة `pending` — شريط رمادي محايد، رسالة "بانتظار وقت رفع التكميل (٩:٠٠ ص)"، وعرض المراكز كقائمة بسيطة بلا حكم تأخير ولا "اكتمل". **حسم غموض الوصف الأصلي:** عدّاد `X / 2` يعكس الرفع الفعلي حتى لو كان مبكرًا (فلو رفع م22 ٧:٣٢ يصبح 1/2)، لكن لا يُطبَّق أي لون نجاح/تأخير ولا وسم "الأسرع" قبل الموعد — لأن الحكم على الالتزام لا يبدأ إلا عند 09:00. (هذا أصحّ منطقيًّا من إجبار 0/2 الوارد كمثال في الوصف.)
  - **بعد 09:00:** كل مركز: مرفوع → ✅ + الوقت؛ غير مرفوع → ⏳ + "متأخر X دقيقة/ساعة" (= الآن − 09:00).
  - العدّ والنسبة: `submitted / 2`.
  - لون الحالة الكلية: 2/2 أخضر "اكتمل التكميل اليوم ✅" · 1/2 أصفر "بانتظار مركز واحد" · 0/2 أحمر "لم يرفع أي مركز التكميل".
  - عند 2/2: وسم ⚡ "(الأسرع)" لأبكر مركز.
  - تدرّج لون المركز المتأخر: < 30 دقيقة أصفر · 30–60 دقيقة برتقالي · > 60 دقيقة أحمر.
  - **حالة استثنائية:** قائمة مراكز فارغة → بطاقة معلومات "لا توجد مراكز مرتبطة بحسابك / الرجاء التواصل مع مدير النظام".
- **التحديث:** `setInterval` كل 60 ثانية يعيد حساب `now` فقط (الحالة تُشتق من الساعة؛ البيانات ثابتة) — بلا وميض. يحقق "كل دقيقة" في الوصف.
